    VBScript using a PB COM DLL

    5 Aug 2010, 06:04 PM
    The thread Need complete, simple example of a PB/Win COM DLL with calling VBSCRIPT code left me questioning the conclusions with regard ByVal and ByRef parameter passing ( 'ByVal varname as Variant' & 'ByRef varname As String' ) and the statement "VBSCRIPT passes object parameters using BYREF".

    I had read that "If a parameter of an object's method is ByVal then VBSript will automatically handle any type conversions necessary."

    So, 'ByVal sText As String' will see the variant sText coerced into a string. PB will then receive a string; albeit unicode.

    I had also read that "If the parameter has been defined as ByRef then VBScript won't do the conversion automatically and expects you as the VBScripter to know what the type is and convert (or Cast) to that type."

    So, '[ByRef] sText As String' should have sText coerced in VBScript by CStr(sText).

    However, this ByRef 'solution' was with regard a Visual Basic component. It does not seem to be required for a PB component.

    As an experiment I wrote the following:
    Code:
    #Compile Dll "VBCallPB"
#Dim All
#Register None
 
#Resource "VBCallPB.pbr"
 
#Com Name "VBCallPBTLB", 1.0
#Com Guid Guid$("{B5FCEC03-E862-4397-B836-9B44910AB15C}")
'#Com TLIB On
 
#Include "WIN32API.inc"
 
Class VBCallPBCom Guid$("{E42268A1-4892-4B47-9E81-4246398BDC1C}") As COM
 
  Interface VBCallPBTLB Guid$("{7C636D90-3CCF-4983-9B55-8ED157E79077}")
    Inherit IDispatch
 
    ' Test 1
    Method ResultValVar (ByVal sText1 As String, ByVal sText2 As String ) As Variant
      sText1 = ACode$(sText1)
      sText2 = ACode$(sText2)
      Method = sText1 + sText2
      sText1 = UCode$(sText1 + "Smith")
    End Method
 
    ' Test 2
    Method ResultValStr (ByVal sText1 As String, ByVal sText2 As String ) As String
      Method = sText1 + sText2
      sText1 = sText1 + UCode$("Smith")
    End Method
 
    ' Test 3
    Method ResultRefVar (sText1 As String, sText2 As String ) As Variant
      sText1 = ACode$(sText1)
      sText2 = ACode$(sText2)
      Method = sText1 + sText2
      sText1 = UCode$(sText1 + "Smith")
    End Method
 
    ' Test 4
    Method ResultRefStr (sText1 As String, sText2 As String ) As String
      Method = sText1 + sText2
      sText1 = sText1 + UCode$("Smith")
    End Method
 
  End Interface
 
End Class
    where VBCallPB.rc is
    Code:
    #include "resource.h"
1 typelib VBCALLPB.TLB
    The VBScript for testing is as follows
    Code:
    Set Test = CreateObject("VBCallPBCom")
 
Text2 = "Roberts"
Text1 = "David"
 
' Test 1
MsgBox Test.ResultValVar( Text1, Text2 ), , "VBScript"
MsgBox Text1, , "VBScript"
 
' Test 2
Text1 = "David"
MsgBox Test.ResultValStr( Text1, Text2 ), , "VBScript"
MsgBox Text1, , "VBScript"
 
' Test 3
Text1 = "David"
MsgBox Test.ResultRefVar( Text1, Text2 ), , "VBScript"
MsgBox Text1, , "VBScript"
 
' Test 4
Text2 = "Roberts" ' Text2 was altered in last test
Text1 = "David"
MsgBox Test.ResultRefStr( Text1, Text2 ), , "VBScript"
MsgBox Text1, , "VBScript"
    Each test uses two message boxes: The first to test the method's value; the second to test a parameter alteration. I used two message boxes rather than one with two lines to avoid a 'nonsense' method value corrupting the parameter alteration; which, of course, may be 'nonsense' as well.

    The first two tests gave the same results:

    DavidRoberts
    David

    and the last two tests also gave the same results:

    DavidRoberts
    DavidSmith

    Note, however, that the inner code for Tests 1 and 3 are the same; the method returning a Variant. The inner code for Tests 2 and 4 are the same; the method returning a String.

    I can explain, for the most part, what is going on. Some parts are not that clear.

    I won't comment yet but let you 'play' and look forward to your comments.

    Don't be fooled by the code into thinking I know what I'm doing - most of you guys visiting the Objects and COM forums can eat me for breakfast.

    I've just added to the PB code:
    Code:
    ' Test 5
Method ResultValVarVar ( ByVal sText1 As Variant, ByVal sText2 As Variant ) As Variant
  Method = Variant$(sText1) + Variant$(sText2)
  sText1 = Variant$(sText1) + "Smith" ' Note that Variant$ coerces to ANSI
End Method
    
' Test 6
Method ResultRefVarStr ( sText1 As Variant, sText2 As Variant ) As String
  Method = UCode$( Variant$(sText1) + Variant$(sText2) )
  sText1 = Variant$(sText1) + "Smith" ' Note that Variant$ coerces to ANSI
End Method
    We are getting a lot of messages now so they have been numbered.
    Code:
    Set Test = CreateObject("VBCallPBCom")

Text2 = "Roberts"
Text1 = "David"

MsgBox Text1, , "VBScript[3]"

    We could have added another four tests but as we have seen the ByVal and ByRef inner code is the same for the same return value type.

    So, we can have 'ByVal sText as String', '[ByRef] sText as String', 'ByVal sText as Variant' and '[ByRef] sText as Variant'.

    In other words, the 'full monty'.

    Other than proving a point, some of them seem to be quite useless in a PB environment.
